# Tone and Voice

## The voice

Write like a knowledgeable colleague explaining something face-to-face — warm, direct, never condescending. The reader is capable; they just haven't used this software yet.

## Dos

- **Use "you"** as the subject: "You can reset your password from the Login screen."
- **Active voice**: "Click **Save**" not "The Save button should be clicked."
- **Present tense**: "The dashboard shows your recent activity" not "The dashboard will show..."
- **Concrete nouns**: name the specific button, field, or screen element exactly as it appears in the UI (bold it on first mention per screen).
- **Short sentences**: aim for ≤ 20 words. Break longer ones.
- **Numbered lists for sequences**: any action that requires more than one step must be a numbered list, not a paragraph.
- **Tip boxes** for non-obvious shortcuts or warnings: `> **Tip:** ...` or `> **Note:** ...`

## Don'ts

- Never use "simply", "just", "easily", "obviously" — these are condescending.
- Never say "click here" — always name what is being clicked.
- Avoid passive voice unless there is no clear actor.
- Don't use technical jargon (API, endpoint, schema) in end-user manuals. If unavoidable, define it.
- Don't assume the user can see the screen you're describing — always tell them where to find it first.

## Vocabulary

| Instead of | Use |
|------------|-----|
| "Navigate to" | "Go to" or "Open" |
| "Click on" | "Click" |
| "Input" (noun) | "field" |
| "Submit the form" | "Click **[button name]**" |
| "Enter your credentials" | "Type your email and password" |
| "Utilize" | "Use" |
